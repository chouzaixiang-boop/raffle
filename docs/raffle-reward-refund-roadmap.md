# 抽奖 / 发奖 / 退款 改造 TODO 与 Postman 测试计划

## 目标

把当前抽奖流程改造成一个更小事务、更高并发、更易测试的方案：

- 抽奖阶段：Redis 负责库存快速扣减，MySQL 只做小事务落库
- 发奖阶段：使用 Redis Stream 异步消费，更新任务状态
- 退款阶段：只允许已发奖任务退款，使用幂等 + 乐观锁 + 配额控制
- 补偿阶段：定时扫描异常任务，修复消息丢失或消费失败

## 当前实现进度

- 模块 1 已完成：抽奖结果落库已拆分为独立事务服务 `RaffleDrawTransactionService`
- 模块 2 已完成：Redis 库存扣减保持原子 `DECR`，并保留回滚保护
- 模块 3 已完成：已新增 `award_task` 任务表的领域对象、仓储、MyBatis mapper，并在抽奖成功时创建 `PENDING` 任务
- 当前可用验证接口：`POST /api/raffle/draw`、`GET /api/raffle/tasks/{taskId}`、`GET /api/raffle/stock/current?strategyId=...&awardId=...`
- 下一步：接入 Redis Stream 发奖消费

## 一、实现顺序 TODO

### 模块 1：抽奖最小事务改造

目标：把抽奖链路收缩到最小事务边界，避免把 Redis、MySQL、Stream 全塞进一个大事务里。

要做的事：

1. 把库存扣减独立到 Redis 原子操作中。
2. 抽奖主流程里只保留 MySQL 小事务：
   - 写 raffle_record
   - 写 award_task
   - 写 outbox_event（如果采用 outbox）
3. 抽奖响应先返回任务结果，不在请求内做发奖。
4. 保证抽奖接口的幂等性，避免重复请求造成重复落库。

Postman 测试点：

- 单次抽奖能成功返回
- 库存为 0 时返回库存不足
- 连续重复请求不会生成重复任务
- 抽奖接口响应时间明显下降

---

### 模块 2：Redis 库存扣减

目标：让库存扣减变成高性能、原子的 Redis 操作。

要做的事：

1. 设计库存 key 规范。
2. 使用 原子命令完成扣减。
3. 处理库存不足和扣减失败的返回值。
4. 与 MySQL 库存写回策略区分开，避免双写冲突。

Postman 测试点：

- 多次请求同一个奖品，库存会逐步减少
- 库存不足后，接口稳定返回失败
- 并发请求下不会出现库存扣成负数

---

### 模块 3：抽奖任务表 award_task

目标：用任务表记录抽奖后的发奖状态，作为后续发奖和退款的核心依据。

要做的事：

1. 创建 award_task 表。
2. 初始化状态流转：PENDING -> PROCESSING -> AWARDED / FAILED。
3. 增加 version 字段支持乐观锁。
4. 给状态字段建立索引，方便扫描和补偿。

Postman 测试点：

- 抽奖后能查到对应 task
- task 初始状态是 PENDING
- 状态修改符合预期

---

### 模块 4：Redis Stream 发奖

目标：用 Redis Stream 替代轮询式定时扫描作为主发奖通道。

要做的事：

1. 设计 Stream key 和 consumer group。
2. 抽奖完成后向 Stream 写入任务消息。
3. 发奖消费者从 Stream 拉取消息并处理。
4. 消费成功后执行 XACK。
5. 失败消息进入 pending，后续由补偿任务处理。

Postman 测试点：

- 抽奖后 Stream 中可以看到任务消息
- 发奖消费者处理后任务状态变为 AWARDED
- 重复消费不会重复发奖

---

### 模块 5：发奖状态机

目标：把发奖逻辑收敛成清晰状态机，避免退款时状态混乱。

要做的事：

1. 统一定义状态枚举。
2. 发奖成功后写入 award_received。
3. 发奖失败时记录失败原因。
4. 状态修改必须带 version 或状态条件。

Postman 测试点：

- 发奖成功后状态变为 AWARDED
- 发奖失败后状态变为 FAILED
- 非法状态不会被错误覆盖

---

### 模块 6：退款幂等与配额控制

目标：限制同一用户在同一活动下的退款次数，并防止重复退款。

要做的事：

1. 建立退款记录表 raffle_refund。
2. 为 userId + activityId 建立退款配额。
3. 使用 refundId 或业务唯一键做幂等控制。
4. 退款前先检查 task_status 是否为 AWARDED。

Postman 测试点：

- 未发奖任务不能退款
- 同一任务重复退款只能成功一次
- 超过配额后拒绝退款

---

### 模块 7：退款库存回补

目标：退款成功后安全地恢复库存，防止超卖。

要做的事：

1. 退款成功后恢复 Redis 库存。
2. MySQL 库存回补使用乐观锁或条件更新。
3. 恢复库存时校验不超过初始库存。
4. 退款与任务状态更新放在同一个小事务内。

Postman 测试点：

- 退款后库存增加
- 库存不会超过初始值
- 并发退款下不会重复加库存

---

### 模块 8：补偿任务与监控

目标：修复 Stream 消息丢失、消费者异常、任务卡住等情况。

要做的事：

1. 定时扫描长时间停留在 PENDING/PROCESSING 的 task。
2. 对失败消息重新投递到 Stream。
3. 增加关键指标：抽奖耗时、发奖耗时、退款耗时、消费堆积量。
4. 增加日志，方便排查任务状态流转问题。

Postman 测试点：

- 模拟消费者异常后，任务能被补偿重新处理
- 模拟失败后，任务能进入 FAILED 并可重试

## 三、每个阶段的 Postman 测试流程

### 阶段 A：抽奖基础流程

#### 1. 准备数据

- 确认活动存在
- 确认策略存在
- 确认奖品库存大于 0

#### 2. 请求接口

- Method: POST
- URL: /api/raffle/draw
- Body:
  - userId
  - strategyId

#### 3. 预期结果

- 返回成功
- 生成 raffle_record
- 生成 award_task
- 库存正确减少

#### 4. 验证点

- 连续调用同一用户，结果是否符合预期
- 库存不足时返回失败
- 响应时间是否明显小于原始大事务版本

---

### 阶段 B：发奖异步流程

#### 1. 先执行一次抽奖

- 让系统生成 PENDING 的 award_task

#### 2. 验证 Stream

- 确认消息已经写入 Redis Stream

#### 3. 触发消费者

- 等待消费者处理，或手动触发消费逻辑

#### 4. 预期结果

- task 状态从 PENDING 变为 AWARDED
- award_received 有记录
- Stream 消息被确认消费

---

### 阶段 C：退款流程

#### 1. 准备前置条件

- 任务状态必须是 AWARDED
- 当前用户退款配额未用尽

#### 2. 请求接口

- Method: POST
- URL: /api/raffle/refund
- Body:
  - userId
  - taskId
  - refundId

#### 3. 预期结果

- 退款成功
- task 状态变为 REFUNDED
- 库存增加
- 退款配额减少

#### 4. 幂等性验证

- 用同一个 refundId 连续请求两次
- 第二次应返回重复请求，不重复退款

#### 5. 配额验证

- 连续退款直到配额用完
- 再次退款应返回“配额不足”

---

### 阶段 D：异常补偿流程

#### 1. 模拟消费者失败

- 让消费者暂停或模拟异常

#### 2. 预期结果

- task 停留在 PENDING 或 PROCESSING
- 补偿任务能够重新拉起处理

#### 3. 退款异常验证

- 模拟退款数据库异常
- 检查 Redis 配额是否被回滚
- 检查库存是否没有错误增加

## 五、建议的验收标准

### 抽奖

- 抽奖接口可以在高并发下稳定返回
- 库存不会变成负数
- 抽奖记录和任务记录都能正确生成

### 发奖

- 发奖与抽奖解耦
- Stream 消息可追踪
- 消费失败可恢复

### 退款

- 只有已发奖任务可退款
- 同一任务不能重复退款
- 同一用户退款次数有限制
- 退款后库存和配额都正确变化
